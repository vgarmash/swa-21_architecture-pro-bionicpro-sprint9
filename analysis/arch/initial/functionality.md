# Функциональность системы BionicPro

## Основные функции

### 1. Управление клиентами (CRM)

Система управления взаимоотношениями с клиентами для учета и управления информацией о пользователях протезов.

**Функции:**
- Хранение персональных данных клиентов
- Управление контактной информацией
- Демографический анализ (возраст, пол, страна)
- Поиск и фильтрация клиентов

**Сущности:**
- Идентификатор клиента (id)
- Имя (name)
- Электронная почта (email)
- Возраст (age)
- Пол (gender)
- Страна (country)
- Адрес (address)
- Телефон (phone)

### 2. Мониторинг протезов (IoT)

Система сбора и анализа данных с EMG (электромиографических) датчиков протезов в реальном времени.

**Функции:**
- Сбор данных с сенсоров протезов
- Мониторинг мышечной активности
- Анализ типов протезов
- Временной анализ сигналов

**Параметры мониторинга:**
- Тип протеза (prosthesis_type)
- Мышечная группа (muscle_group)
- Частота сигнала (signal_frequency)
- Длительность сигнала (signal_duration)
- Амплитуда сигнала (signal_amplitude)
- Время регистрации (signal_time)

### 3. Генерация отчетов

Система формирования отчетов об использовании протезов.

**Функции:**
- Генерация отчетов об использовании
- Экспорт данных в различных форматах
- Фильтрация по пользователям
- Аналитика и статистика

**Доступ к отчетам:**
- Пользователи с ролью `prothetic_user` могут загружать отчеты
- Аутентификация через Keycloak
- Авторизация на основе ролей

### 4. Аутентификация и авторизация

Централизованная система управления доступом.

**Функции:**
- Аутентификация пользователей (логин/пароль)
- Управление ролями
- OAuth 2.0 / OpenID Connect
- Интеграция с LDAP

**Роли пользователей:**
| Роль | Описание | Права |
|------|----------|-------|
| user | Обычный пользователь | Базовый доступ |
| administrator | Администратор | Полный доступ к системе |
| prothetic_user | Пользователь протеза | Доступ к отчетам |

### 5. Интеграция с LDAP

Централизованное управление пользователями через LDAP-каталог.

**Функции:**
- Синхронизация пользователей
- Управление группами
- Аутентификация через LDAP

**Группы LDAP:**
- cn=user - обычные пользователи
- cn=prothetic_user - пользователи протезов

## Пользовательские сценарии

### Сценарий 1: Аутентификация пользователя

1. Пользователь открывает веб-приложение
2. Система перенаправляет на Keycloak для аутентификации
3. Пользователь вводит учетные данные
4. Keycloak возвращает токен доступа
5. Приложение сохраняет токен и предоставляет доступ

### Сценарий 2: Загрузка отчета

1. Аутентифицированный пользователь нажимает "Download Report"
2. Приложение отправляет запрос к API с токеном Bearer
3. API проверяет права доступа
4. При положительной проверке данные извлекаются из OLAP базы
5. Отчет возвращается пользователю

### Сценарий 3: Управление данными клиентов

1. Администратор аутентифицируется в системе
2. Администратор получает доступ к CRM данным
3. Система выполняет CRUD операции с PostgreSQL
4. Данные сохраняются и индексируются

### Сценарий 4: Мониторинг протезов

1. Датчики протеза собирают EMG данные
2. Данные передаются в систему
3. ClickHouse сохраняет данные для аналитики
4. Система анализирует паттерны и аномалии

## Технологический стек

### Фронтенд
- React 18
- TypeScript
- Tailwind CSS
- React Keycloak
- Nginx (проксирование)

### Базы данных
- PostgreSQL 14 (CRM)
- ClickHouse (OLAP)

### Аутентификация
- Keycloak 21
- OpenLDAP

### Инфраструктура
- Docker
- Docker Compose
- MinIO (объектное хранилище)

## Ограничения и требования

### Требования к безопасности
- Все API должны использовать HTTPS в продакшне
- Токены доступа должны истекать через ограниченное время
- Пароли должны хешироваться и храниться безопасно

### Требования к производительности
- OLAP запросы должны выполняться не более 5 секунд
- Frontend должен загружаться менее 3 секунд
- Система должна поддерживать 100+ одновременных пользователей

### Требования к доступности
- Время работы системы: 99.9%
- Резервное копирование баз данных: ежедневно
- Логирование всех операций
