## Code Audit Report for BionicPRO Architecture

**Date:** 2026-02-18

This report details a comprehensive audit of the BionicPRO codebase within the `app/` directory, comparing its implementation against technical specifications provided in `task1/impl/` and `task2/impl/` markdown files. The audit identifies discrepancies, logical errors, security vulnerabilities, and code quality issues, categorized by criticality.

---

### 1. `bionicpro-auth` Service Audit Findings

**Критические проблемы (Critical):**

1.  **Непостоянный/Невосстановимый ключ шифрования и жестко закодированная соль в `OAuth2ClientConfig.java`**:
    *   **Описание**: `BytesEncryptor` использует случайный UUID в качестве секрета шифрования при каждом запуске приложения и жестко закодированную "соль". Это предотвращает расшифровку данных (например, токенов обновления), зашифрованных предыдущими экземплярами после перезапуска приложения, что приводит к принудительным повторным входам и потере сеанса. Жестко закодированная соль еще больше ослабляет шифрование.
    *   **Рекомендация**: Ключ шифрования и соль ДОЛЖНЫ быть безопасно сгенерированы один раз, храниться внешне (например, переменные среды, управление секретами) и оставаться постоянными при перезапусках и экземплярах приложения. Жестко закодированная "соль" должна быть заменена безопасно сгенерированным, уникальным значением.

2.  **Недостаточная ротация сеансов в `SecurityConfig.java` и `SessionRotationFilter`**:
    *   **Описание**: `SessionRotationFilter` ротирует сеансы только на `/api/auth/status` и `/api/auth/refresh` конечных точках. ТЗ требует ротацию сеансов "При каждом успешном запросе" для защиты от атак фиксации сеанса.
    *   **Рекомендация**: Изменить `SessionRotationFilter`, чтобы он запускал `sessionService.rotateSession()` для *всех* аутентифицированных запросов. Альтернативно, использовать встроенные механизмы защиты от фиксации сеанса Spring Session.

3.  **Глобально отключена защита CSRF в `SecurityConfig.java`**:
    *   **Описание**: Защита CSRF отключена во всех `SecurityFilterChain` бинах. Для аутентификации на основе сеансов с использованием файлов cookie HTTP-only защита CSRF имеет решающее значение для операций, изменяющих состояние.
    *   **Рекомендация**: Включить защиту CSRF, особенно для `apiProxySecurityFilterChain` и `defaultSecurityFilterChain`. Если конкретные конечные точки не требуют CSRF, их следует явно исключить.

4.  **Отсутствует отзыв токена обновления Keycloak при выходе в `AuthController.java` и `SessionService.java`**:
    *   **Описание**: Метод `logout()` и `invalidateSession()` только аннулируют локальный BFF сеанс и очищают файл cookie. Токен обновления, хранящийся в Keycloak, остается действительным, что позволяет использовать скомпрометированный токен для получения новых токенов доступа.
    *   **Рекомендация**: Реализовать механизм отзыва токена обновления в конечной точке отзыва токенов Keycloak, когда пользователь выходит из BFF.

5.  **Автоматическое обновление токенов не реализовано (TODO) в `SessionService.java`**:
    *   **Описание**: Метод `validateAndRefreshSession()` содержит комментарий `TODO`, указывающий на отсутствие логики вызова Keycloak для обновления токенов. Это вынудит пользователей повторно аутентифицироваться после истечения срока действия токена доступа (2 минуты).
    *   **Рекомендация**: Реализовать логику вызова конечной точки токенов Keycloak (`tokenUri`), используя `refreshToken`, для получения нового токена доступа (и, возможно, нового токена обновления) и обновления `SessionData`.

6.  **Проблема масштабируемости: Хранилище `authRequestStore` в памяти для параметра `state` в `SessionService.java`**:
    *   **Описание**: `authRequestStore` использует `ConcurrentHashMap` для хранения параметров `state` во время потока входа. Это проблематично в горизонтально масштабируемых развертываниях, так как запросы на `login` и `callback` могут попадать в разные экземпляры, что приводит к несоответствиям `state` и сбоям аутентификации.
    *   **Рекомендация**: Это хранилище `authRequestStore` должно быть поддержано распределенным хранилищем, таким как Redis. Spring Security `OAuth2AuthorizationRequestRepository` должен обрабатывать это прозрачно, если `HttpSession` поддерживается Redis.

7.  **Отсутствует секрет клиента Keycloak в `application.yml`**:
    *   **Описание**: В `application.yml` отсутствует свойство `keycloak.client-secret`, которое требуется для аутентификации службы `bionicpro-auth` в Keycloak в качестве конфиденциального клиента.
    *   **Рекомендация**: Добавить `client-secret: ${KEYCLOAK_CLIENT_SECRET}` в `application.yml` в раздел `keycloak:`.

**Мелкие проблемы / Предложения:**

1.  **Несогласованность атрибута `SameSite` в файлах cookie**: `RedisConfig.java` устанавливает `SameSite=Lax`, тогда как `task1/impl/REPORT.md` указывает `SameSite=Strict`.
    *   **Рекомендация**: Уточнить желаемую политику `SameSite` и обеспечить согласованность. `Strict` обычно рекомендуется для более надежной безопасности, если это совместимо с потоками пользовательского интерфейса.
2.  **`username` в `AuthStatusResponse`**: `AuthController.getStatus()` использует `userId` в качестве `username` вместо `preferred_username` из токена ID (который хранится в `SessionData`).
    *   **Рекомендация**: Получить `username` из `SessionData`, используя `preferred_username`.
3.  **Избыточная авторизация `/api/auth/**` в `apiProxySecurityFilterChain` (`SecurityConfig.java`)**: Может сбивать с толку, поскольку `authSecurityFilterChain` обрабатывает эти пути.
    *   **Рекомендация**: Удалить избыточное правило авторизации.
4.  **Рефакторинг внедрения `SecurityContextRepository` (`SecurityConfig.java`)**: Незначительно, так как Spring Security управляет им внутренне.
5.  **Отсутствует поле `codeVerifier` в `SessionData.java`**: Хотя Spring Security может обрабатывать его внутренне в HTTP-сессии, если `SessionService` предназначен для явного управления состоянием PKCE, это поле потребуется.
    *   **Рекомендация**: Проверить, достаточно ли внутренней обработки Spring Security.

### 2. `bionicpro-reports` Service Audit Findings

**Критические проблемы (Critical):**

1.  **Отсутствует непривилегированный пользователь в `Dockerfile`**:
    *   **Описание**: Dockerfile не создает и не переключается на непривилегированного пользователя, запуская контейнер от имени root.
    *   **Рекомендация**: Создать непривилегированного пользователя и запускать приложение от имени этого пользователя в Dockerfile.

2.  **Жестко закодированный `connectionTimeout` в `ClickHouseConfig.java`**:
    *   **Описание**: `connectionTimeout` загружается из `application.yml` как строка, но затем игнорируется, так как `dataSource.setConnectionTimeout(2000)` жестко закодировано. Это предотвращает правильную настройку критически важного параметра пула соединений.
    *   **Рекомендация**: Правильно разобрать внедренную строку `connectionTimeout` и применить ее к `HikariDataSource.setConnectionTimeout()`.

3.  **Чрезмерно разрешительный `anyRequest().permitAll()` в `SecurityConfig.java`**:
    *   **Описание**: Конфигурация безопасности заканчивается `anyRequest().permitAll()`, что означает, что любая конечная точка, не соответствующая явно, будет доступна без аутентификации.
    *   **Рекомендация**: Изменить `anyRequest().permitAll()` на `anyRequest().authenticated()` или `anyRequest().denyAll()` и явно внести в белый список публичные конечные точки для защиты всех путей по умолчанию.

4.  **`UserReport` Модель не соответствует схеме ClickHouse (`UserReport.java`)**:
    *   **Описание**: Модель сущности `UserReport`, используемая в службе Java, не соответствует фактической схеме таблицы `user_reports`, заполняемой DAG ETL Airflow в ClickHouse. В ней отсутствуют аналитические поля (например, `total_sessions`, `avg_signal_amplitude`) и содержатся общие поля отчета.
    *   **Рекомендация**: Обновить `UserReport.java` для точного соответствия схеме таблицы `user_reports`, определенной в `task2/impl/02_etl_airflow_dag.md`.

5.  **`ReportResponse` DTO Структура не соответствует (`ReportResponse.java`)**:
    *   **Описание**: Реализованный DTO `ReportResponse` не соответствует подробной структуре, предложенной в ТЗ (`task2/impl/03_reports_api_service.md`) для агрегированных аналитических данных, включая объект `CustomerInfo`.
    *   **Рекомендация**: Изменить `ReportResponse.java` для соответствия структуре, определенной в `task2/impl/03_reports_api_service.md`, добавив агрегированную статистику и `CustomerInfo`.

6.  **`ReportRepository` Запрос/Маппер не соответствует схеме ClickHouse (`ReportRepository.java`)**:
    *   **Описание**: `userReportRowMapper` и все SQL-запросы в репозитории ожидают столбцы, отсутствующие в фактической таблице `user_reports` ClickHouse. Это приведет к ошибкам во время выполнения (например, `SQLException: Column 'id' not found`).
    *   **Рекомендация**: После исправления `UserReport.java`, обновить `userReportRowMapper` и все SQL-запросы для выбора и сопоставления правильных столбцов из фактической таблицы `user_reports` ClickHouse. Пересмотреть методы, полагающиеся на столбец `id`.

7.  **`ReportService` Работает с неправильными моделями**:
    *   **Описание**: Основная бизнес-логика нарушена, потому что она использует неправильные модели данных.
    *   **Рекомендация**: См. рекомендации для `UserReport.java` и `ReportResponse.java`.

8.  **Несогласованный тип ID пользователя (`String` против `Long`/`UInt32`) в `ReportController.java` и `ReportRepository.java`**:
    *   **Описание**: `userId` передается как `String` из `jwt.getSubject()` в методы `ReportService` и `ReportRepository`, но соответствующий столбец в ClickHouse (`user_id`) имеет тип `UInt32`. Это приводит к несоответствию типов и потенциальным проблемам с извлечением и фильтрацией данных.
    *   **Рекомендация**: Реализовать вспомогательный метод `extractUserId(Jwt jwt)` в `ReportController` для преобразования субъекта JWT в `Long` (или `Integer`) и использовать этот числовой тип последовательно в `ReportService` и `ReportRepository` для операций с базой данных.

**Мелкие проблемы / Предложения:**

1.  **Отсутствует таймаут чтения/записи ClickHouse**: Хотя таймаут пула соединений установлен, явные таймауты чтения/записи для драйвера JDBC ClickHouse не настроены.
    *   **Рекомендация**: Рассмотреть возможность добавления параметров `socketTimeout` и `connectTimeout` к URL JDBC ClickHouse.
2.  **Отсутствует проверка работоспособности в `Dockerfile`**: Хорошая практика для оркестраторов.

### 3. Результаты аудита DAG Airflow

**Критические проблемы (Critical):**

1.  **Жестко закодированные учетные данные базы данных в DAG (`bionicpro_etl_dag.py`)**:
    *   **Описание**: Учетные данные базы данных для `sensors-db` и `crm-db` жестко закодированы непосредственно в скрипте Python для `extract_sensors_data` и `extract_crm_data`.
    *   **Рекомендация**: Удалить жестко закодированные учетные данные. Использовать Airflow Connections (безопасно настроенные в пользовательском интерфейсе Airflow или через бэкэнд секретов) и `airflow.providers.postgres.hooks.postgres.PostgresHook` для получения сведений о подключении.

**Мелкие проблемы / Предложения:**

1.  **Учетные данные `clickhouse_driver`**: Если ClickHouse требует аутентификации, эти учетные данные также должны безопасно управляться через Airflow Connections.
    *   **Рекомендация**: Если для ClickHouse требуется аутентификация, получить учетные данные из Airflow Connections.
2.  **Отсутствующие импорты в `load_to_olap`**: `pandas` и `clickhouse_driver` не импортируются явно внутри функции `load_to_olap`, что требуется при использовании `PythonOperator`.
    *   **Рекомендация**: Добавить `import pandas` и `from clickhouse_driver import Client` внутри функции `load_to_olap`.

### 4. Результаты аудита фронтенд-сервиса

**Критические проблемы (Critical):**

1.  **Wildcard `Access-Control-Allow-Origin: '*'` в `nginx.conf`**:
    *   **Описание**: Конфигурация Nginx разрешает междоменные запросы с *любого* домена. Это серьезная уязвимость безопасности в производственной среде, так как она обходит политику одинакового происхождения браузера.
    *   **Рекомендация**: В производственной среде заменить `'*'` на конкретные, разрешенные источники (например, `'https://your-domain.com'`).

**Основные проблемы (Major):**

1.  **Отсутствует конфигурация PKCE в `App.tsx`**:
    *   **Описание**: Объект `keycloakConfig` не включает `pkceMethod: 'S256'` и `flow: 'standard'`, как того требует ТЗ (`task1/impl/02_task2_pkce.md`). Это означает, что PKCE неявно не настроен во фронтенде, что подрывает ключевое требование безопасности.
    *   **Рекомендация**: Добавить `pkceMethod: 'S256'` и `flow: 'standard'` в объект `keycloakConfig` в `App.tsx`.

**Мелкие проблемы / Предложения:**

1.  **Отсутствует `REACT_APP_AUTH_URL` в `.env`**: В файле `.env` отсутствует переменная, предложенная ТЗ для URL BFF.
    *   **Рекомендация**: Добавить `REACT_APP_AUTH_URL=http://localhost:8000` в `.env`.
2.  **Устаревшая версия Node.js в `Dockerfile`**: Node.js 16 приближается к концу срока службы.
    *   **Рекомендация**: Обновить базовый образ Node.js до более новой версии LTS (например, 18 или 20).
3.  **Нет непривилегированного пользователя в `Dockerfile`**: Dockerfile фронтенда явно не запускается от имени непривилегированного пользователя.
    *   **Рекомендация**: Явно создать и переключиться на непривилегированного пользователя в Dockerfile для лучшей безопасности.
4.  **Нет проверки работоспособности в `Dockerfile`**:
    *   **Рекомендация**: Добавить инструкцию `HEALTHCHECK` в Dockerfile.
5.  **`target: "es5"` в `tsconfig.json`**:
    *   **Рекомендация**: Рассмотреть возможность установки `target` на `es2015` или `es2017` для лучшего соответствия современным функциям JavaScript.
6.  **Избыточный вызов `fetch` в `downloadReport()` (`ReportPage.tsx`)**: Функция `downloadReport` повторно извлекает данные, потенциально дублируя сетевой вызов, если отображаемые данные актуальны.
    *   **Рекомендация**: Оптимизировать `downloadReport` для использования существующего состояния, если оно считается актуальным, или уточнить намерение всегда извлекать самые последние данные.
7.  **Массив зависимостей `useEffect` для `fetchReport` (`ReportPage.tsx`)**: `fetchReport` является зависимостью, что может привести к ненужным повторным созданиям.
    *   **Рекомендация**: Обернуть `fetchReport` в `useCallback`.

### 5. Результаты аудита `docker-compose.yaml` и инициализации БД

**Критические проблемы (Critical):**

1.  **Жестко закодированные учетные данные и пустой `FERNET_KEY` для Airflow (`docker-compose.yaml`)**:
    *   **Описание**: `airflow-webserver` имеет `AIRFLOW__CORE__FERNET_KEY: ''` и жестко кодирует учетные данные базы данных в `AIRFLOW__CORE__SQL_ALCHEMY_CONN`.
    *   **Рекомендация**: Сгенерировать и безопасно хранить ключ Fernet, внедрить его в `AIRFLOW__CORE__FERNET_KEY`. Не жестко кодировать учетные данные базы данных для базы данных метаданных Airflow или любой другой базы данных; использовать систему управления секретами.

**Основные проблемы (Major):**

1.  **Отсутствующие сервисы Airflow Scheduler/Worker/Triggerer (`docker-compose.yaml`)**:
    *   **Описание**: Хотя `airflow-webserver` использует `LocalExecutor` для разработки, надежная и масштабируемая настройка Airflow обычно требует отдельных сервисов `airflow-scheduler`, `airflow-worker` и `airflow-triggerer`. Их отсутствие указывает на функциональный пробел для производственной системы ETL.
    *   **Рекомендация**: Добавить сервисы `airflow-scheduler`, `airflow-worker` и `airflow-triggerer` в `docker-compose.yaml` для полного и масштабируемого развертывания Airflow.

**Мелкие проблемы / Предложения:**

1.  **`bionicpro-auth` `KEYCLOAK_REDIRECT_URI` (`docker-compose.yaml`)**: Использует `http://localhost:8000/api/auth/callback`. Хотя это функционально, для ясности в контексте сети Docker иногда предпочтительнее использовать имя сервиса (`http://bionicpro-auth:8000/...`) для внутренней связи, хотя `localhost` обычно подходит для перенаправлений браузера.
    *   **Рекомендация**: Изменений не требуется, но учитывайте, как этот URL используется браузером по сравнению с внутренней связью в сети Docker.
